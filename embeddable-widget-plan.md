# خطة بناء Embeddable Chat Widget
### الشات كـ فقاعة عايمة جوه موقع التاجر نفسه (زي Intercom / Tawk.to)

الفكرة: التاجر بياخد `<script>` tag واحد، يحطه في كود موقعه، وتظهرله فقاعة شات عايمة تفتح شات بوته من غير ما الزائر يسيب موقع التاجر خالص.

---

## المرحلة 0: افهم المكونات التلاتة الأساسية

النظام ده بيتكوّن من 3 أجزاء منفصلة، ومهم تفرّق بينهم من البداية:

1. **Loader Script** (`widget.js`) — ملف JS صغير جدًا، ده اللي التاجر بيحطه في موقعه
2. **Widget UI** — نفس صفحة الشات بتاعتك (`/chat/[slug]`) بس بتتحمّل جوه `<iframe>`
3. **Bridge/Communication Layer** — الكود اللي بيخلي الـ iframe (اللي جوه موقع التاجر) يتكلم مع صفحتك الأصلية (لفتح/قفل الفقاعة، تغيير الحجم، إلخ)

---

## المرحلة 1: الـ Loader Script

- [ ] اعمل ملف مستقل `public/widget.js` (هيتقدم من نفس الدومين بتاعك، زي `yourplatform.com/widget.js`)
- [ ] الكود ده لازم يكون **vanilla JS خفيف جدًا** (من غير React/framework) عشان أي موقع مهما كان الـ stack بتاعه يقدر يحمّله من غير تعارض
- [ ] وظيفته:
  ```js
  (function() {
    var merchantSlug = document.currentScript.getAttribute('data-merchant');
    var bubble = document.createElement('div');
    // إنشاء زرار الفقاعة العايمة (position: fixed, bottom-right)
    var iframe = document.createElement('iframe');
    iframe.src = 'https://yourplatform.com/widget-frame/' + merchantSlug;
    // الـ iframe مخفي الأول، يظهر لما المستخدم يدوس على الفقاعة
  })();
  ```
- [ ] التاجر هيحط الكود ده في موقعه كده:
  ```html
  <script src="https://yourplatform.com/widget.js" data-merchant="bright-bites-bakery" async></script>
  ```
- [ ] لازم الملف يبقى **immutable/cached بشكل ذكي** — لو غيّرت في المنطق، استخدم versioning في اسم الملف أو query param (`widget.js?v=2`) عشان الكاش القديم عند التجار ميعندش يمنع التحديث

---

## المرحلة 2: صفحة الـ Widget نفسها (جوه الـ iframe)

- [ ] اعمل route منفصل عن صفحة الشات العادية: `app/widget-frame/[merchantSlug]/page.tsx`
- [ ] **ليه route منفصل مش نفس `/chat/[slug]`؟** لأن نسخة الـ widget محتاجة تصميم مختلف شوية (بدون الـ Navbar/Footer بتاع منصتك، وبدون الـ "Discover More" sidebar) — التاجر مش عايز زائر موقعه يشوف حاجات تودّيه بره موقعه
- [ ] الصفحة دي لازم تبقى **minimal**: الشات بس، من غير أي عناصر تانية من منصتك
- [ ] استخدم نفس منطق الـ backend (context builder، endpoints) اللي عملته لصفحة `/chat/[slug]` العادية — الفرق بس في الـ UI مش في المنطق

---

## المرحلة 3: الـ HTTP Headers (أهم جزء تقني وأسهل نقطة تتنسى)

الـ iframe ده هيتحمّل جوه دومين مختلف تمامًا (موقع التاجر)، فلازم تسمحله صراحة:

- [ ] في `next.config.mjs`، اضبط الـ headers لصفحة `widget-frame` بس (مش باقي الموقع):
  ```js
  async headers() {
    return [
      {
        source: '/widget-frame/:path*',
        headers: [
          { key: 'X-Frame-Options', value: '' }, // امسحه أو خليه فاضي عشان الصفحة دي بس
          { key: 'Content-Security-Policy', value: "frame-ancestors *;" }
          // في الإنتاج: بدل * حط قائمة الدومينات المسموحة بتاعة التجار المسجلين
        ]
      }
    ]
  }
  ```
- [ ] **متسيبش `frame-ancestors *` بشكل دايم في production** لو عايز أمان أعلى — الأفضل تتحقق من الدومين وقت الـ request وتسمح بس للدومين اللي التاجر سجّله فعليًا في بياناته (يعني تضيف حقل `allowed_domain` لجدول `Merchant`)
- [ ] باقي صفحات المنصة (الداشبورد، الـ auth) لازم تفضل زي ما هي — **ميتفتحوش في iframe خالص** (`X-Frame-Options: DENY`) عشان الحماية من clickjacking

---

## المرحلة 4: الـ Bridge — التواصل بين الفقاعة والـ iframe

- [ ] استخدم `window.postMessage` للتواصل بين الصفحتين (موقع التاجر ↔ صفحة الـ widget جوه الـ iframe):
  - لما المستخدم يدوس على الفقاعة → الـ loader script يبعت `postMessage` للـ iframe يقوله "افتح"
  - الـ iframe لما يحمّل ويبقى جاهز → يبعت رسالة "أنا جاهز" للصفحة الأم
  - لو المستخدم بعت رسالة وعايز الفقاعة تكبر (عشان في محتوى كتير) → الـ iframe يبعت "غيّر الحجم لـ X"
- [ ] **لازم تتحقق من `event.origin`** في كل مستمع `postMessage` — متستقبلش رسائل من أي مصدر، تأكد إنها جاية من الـ iframe بتاعك فعلاً، مش أي سكريبت خبيث تاني على نفس صفحة التاجر

---

## المرحلة 5: التصميم البصري للفقاعة

- [ ] الفقاعة (زرار الفتح) لازم تكون قابلة للتخصيص بسيط من التاجر (لون، موضع يمين/شمال) — تقدر تعمل ده بـ data attributes زيادة على السكريبت:
  ```html
  <script src="..." data-merchant="..." data-color="#F26B1D" data-position="bottom-right"></script>
  ```
- [ ] الفقاعة والـ iframe لازم يبقوا بـ `z-index` عالي جدًا عشان ميضيعوش وسط عناصر موقع التاجر
- [ ] على الموبايل: الشات المفتوح لازم ياخد الشاشة كلها (fullscreen) مش يفضل صغير في الزاوية

---

## المرحلة 6: التسجيل والتوليد للتاجر

- [ ] في صفحة "Invite Customers to Chat"، أضف **تبويبين**: "Copy Link" (اللي موجود بالفعل) و **"Copy Embed Code"** (الجديد)
- [ ] الكود اللي هيتنسخ يتولد ديناميكيًا بالـ slug بتاع التاجر:
  ```ts
  const embedCode = `<script src="${appUrl}/widget.js" data-merchant="${merchant.slug}" async></script>`
  ```
- [ ] وريله preview بسيط جوه صفحة الداشبورد (iframe صغير يوضحله شكل الفقاعة قبل ما ينسخ الكود)

---

## المرحلة 7: الاختبار

- [ ] جرب الكود فعليًا على صفحة HTML بسيطة (مش على موقعك) — أسهل حاجة تعمل ملف `test.html` فاضي وتحط فيه السكريبت وتفتحه محليًا
- [ ] اختبر على موقع فيه framework تاني خالص (WordPress مثلاً) عشان تتأكد مفيش تعارض CSS
- [ ] اختبر الحالة اللي التاجر بيحط السكريبت غلط أو الـ slug بتاعه غلط — لازم الفقاعة تختفي بهدوء أو توضح رسالة خطأ بسيطة، مش تكسر موقع التاجر كله

---

## ترتيب الأولوية

الفيتشر ده **مش أساسي للـ MVP** — ابدأ بنظام اللينك البسيط (اللي عندك بالفعل في التصميم) وشغّله وجرّبه مع تجار حقيقيين الأول. الـ widget ده فيتشر "premium" منطقي تضيفه بعدين لما تكون عايز تفرّق بين خطط الاشتراك (زي "الباقة المدفوعة بتاخد Widget حقيقي، الباقة المجانية بتاخد لينك بس") — وده كمان بيديك سبب تجاري واضح تربط بيه الـ subscription tiers اللي في خطة العمل بتاعتك أصلاً.
